# LTR bug list

Status legend: [ ] open · [repro] reproduced with a failing test (LtrBugTriageTest) · [fixed] · [?] not yet reproduced

## Original list

- [fixed] **Wizard's Rockets** — no choosing of X. The engine handled a pre-supplied X, but the legal-actions/UI path submits a bare ActivateAbility with no xValue, so X defaulted to 0 and the ability produced no mana. Added an `{X}`-mana-cost pause in `ActivateAbilityHandler` (a ChooseNumberDecision + `ActivateAbilityChooseManaXContinuation`), mirroring the existing TapX pause. Covered by WizardsRocketsXChoiceScenarioTest; the engine-direct path (WizardsRocketsScenarioTest) is unchanged.
- [fixed] **Spiteful Banditry** — Treasure triggers multiple times. Was a per-creature `leavesBattlefield(binding=ANY)` trigger; simultaneous deaths each fired before the once-per-turn marker was set. Fixed by adding a batched, opponent-scoped `Triggers.OneOrMoreCreaturesAnOpponentControlsDie` (the die-batch trigger now honors the filter's controller predicate, mirroring the enter-batch). Covered by SpitefulBanditryScenarioTest.
- [?] **Lobelia Sackville-Baggins** — doesn't trigger. Existing scenario test passes; need a tighter repro (does it fail to go on the stack, or trigger-then-fizzle? how/whose-turn was the opponent's creature put into the graveyard "this turn"?).
- [fixed] **Pippin's Bravery** — with no Food you could still choose "Sacrifice a Food? -> Yes" and wrongly get +4/+4 (nothing was sacrificed). Root cause: "You may sacrifice a Food. If you do…" was modeled as an `OptionalCostEffect` cost-gate, but sacrificing here is a resolution-time optional action, not a cost — and the cost-gate's affordability check failed open for a `SacrificeEffect`, offering "yes" even with no Food. Fixed by remodeling the spell as an `IfYouDoEffect` over a Gather → `ChooseUpTo(1)` → Sacrifice pipeline (cf. Heated Argument), gating the +4/+4 vs +2/+2 branch on `SuccessCriterion.CollectionNonEmpty` over the *chosen* pile: with no Food there's nothing to choose, so no prompt appears and the +2/+2 branch runs. Composes existing primitives — no new SDK type. Covered by PippinsBraveryScenarioTest (no Food -> no prompt, +2/+2; with Food -> accept, +4/+4 and Food sacrificed; with Food -> decline, +2/+2 and Food kept). The buff itself always resolved correctly — the earlier "target fizzles" theory was a test-harness artifact, not engine behavior.

- [fixed] **Ring-bearer preserved between zones** — `RingBearerComponent` persisted across exile→return. Fixed by stripping it in `ZoneMovementUtils.stripBattlefieldComponents` (CR 701.54e / 400.7). Covered by RingBearerAndEquipmentZoneChangeScenarioTest.
- [fixed] **Meneldor — Bilbo's Ring still attached** — equipment whose host leaves the battlefield must fall off (CR 704.5n; Auras → graveyard, 704.5m). The real-pipeline repro (`MeneldorBilbosRingDetachScenarioTest`: Meneldor's combat-damage trigger blinks the equipped host through the stack, then a priority pass) confirmed it was a **genuine** bug, not a false positive: `ZoneTransitionService.moveToZone` reuses the host's EntityId across the exile→battlefield round-trip, so the `UnattachedAurasCheck` SBA — which keyed off "host id no longer on the battlefield" — saw the returned same-id object and left the attachment in place. Fixed *without* re-introducing the reverted eager-detach (which broke Forebears Blade's detection): `moveToZone` now only *marks* the leaving host's attachments with a transient `AttachmentHostLeftComponent` (leaving `AttachedToComponent` intact so the live `aurasByTarget` index still surfaces ATTACHED-binding dies/leaves triggers), and the SBA detaches/graveyards on that marker after triggers are detected. `stripBattlefieldComponents` clears the marker so a returning object never carries a stale one. Covers Equipment and Aura; Forebears Blade re-attach still works.
- [fixed] **Bilbo's Ring — attacks alone.** The "attacks alone" trigger fired even when the equipped creature attacked alongside others. `AttachmentTriggerDetector` (which handles ATTACHED-binding triggers) fired on bare membership in the attacker set without applying the trigger's `requires` predicates (`AttackPredicate.Alone`). Fixed to apply `requires`, mirroring the main loop. Covered by BilbosRingAttacksAloneScenarioTest; empty-`requires` attached-attack triggers (Stormbeacon Blade, Thunder Lasso) still fire.

## Added later

- [not-a-bug] **Glorious Gale** — verified working: the existing GloriousGaleScenarioTest pins both "counters a legendary creature spell and the Ring tempts you" (temptCount 1) and the nonlegendary case (0). The legendary check on the countered spell-on-stack resolves correctly; no change needed.
- [fixed] **Elrond, Master of Healing** — drew a card even when the targeted creature had no +1/+1 counter. `TriggerMatcher.matchesBecomesTargetTrigger` never evaluated the target filter's `statePredicates`; fixed to apply them. Covered by ElrondMasterOfHealingTargetTriggerTest.
- [fixed] **Frodo Baggins** — power/toughness was 1/1, should be 1/3. Fixed in the LTR field-verification pass (commit "Verify LTR card fields against Scryfall; fix P/T and text divergences").
- [fixed] **Elrond, Master of Healing** — scry trigger offered 0 targets ("up to up to X target creatures, 0 / 0"). Two bugs: (1) the target cap `dynamicMaxCount = ContextProperty(TRIGGER_SCRY_COUNT)` resolved to 0 — `TriggerProcessor.snapshotDynamicCount` built the `EffectContext` without `triggerScryCount`, so `DynamicAmountEvaluator` read `context.triggerScryCount ?: 0`. Fixed by passing `triggerScryCount = trigger.triggerContext.scryCount` into that context, so after a Serum Visions (scry 2) the cap is 2. (2) The prompt doubled the quantifier ("up to up to …"): `TargetObject.description` always prepended "up to " for optional targets even when the author-supplied `id` already began with it (21 cards do, e.g. `target("up to X target creatures", …)`). Fixed in the SDK to skip the prefix when the `id` already starts with the quantifier — `description` is computed (not serialized), so no snapshot churn and all such cards benefit. Covered by `ElrondMasterOfHealingScryTargetCountTest`.
- [fixed] **Friendly Rivalry** — could choose the same creature for both "creature you control" targets. The card's second target used `.other()` (which only excludes the spell's source) instead of `TargetOther`, the requirement wrapper the engine already validates as "different from earlier targets this cast" (CR 601.2c). Switched to `TargetOther`. Covered by FriendlyRivalryScenarioTest; snapshot re-blessed.
- [fixed] **The One Ring — protection from everything doesn't prevent combat damage.** A player with The One Ring's "protection from everything" (`Effects.GrantPlayerProtection()` → `PlayerProtectionComponent`) still took combat damage. Root cause: non-combat damage routes through `DamageUtils.dealDamageToTarget`, which consults `PlayerProtectionRules.isProtectedFromSource`, but combat damage is applied by `CombatDamageManager.applyDamageToPlayer`, which reduces life directly and never checked player protection. Fixed by adding a `PlayerProtectionModifier` to the combat-damage Modify phase — it drops any assignment whose target player is protected from the attacking source (using the same `PlayerProtectionRules`), mirroring the keyword `ProtectionModifier` and Deep Wood's `PreventDamageFromAttackingCreaturesModifier`. Honors `isDamagePreventionDisabled` (Fear, Fire, Foes! / Sunspine Lynx) so genuinely unpreventable damage still lands. Covered by `PlayerProtectionDamageScenarioTest` (a creature may attack the protected player but its combat damage is prevented; unprotected control still takes 2). The four official rulings (oracle `3aa83ed2-…`) were added to `TheOneRing.kt`; snapshot re-blessed.
